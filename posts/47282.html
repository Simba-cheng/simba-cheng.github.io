在 ElasticSearch-基础-02-基本概念 中，了解了路由功能。可以看出 elasticsearch 路由机制 与其分片机制有着直接关系。elasticsearch 通过 hash 算法，将具有相同 hash 值的文档分配到同一分片中。这个 hash 算法和负载均衡有点类似。

默认情况，es使用 文档id作为hash值，然后对其hash、分配。这种做法的好处是，不会有数据热点(数据倾斜)

1. 为什么要自定义 routing 值？

在数据量、主分片数不大的情况下，常规用doc_id作为routing值，已经能满足’平均的分布数据’的需求。

但如果数据量非常庞大，设置了很多主分片数(分片数)。这时，一个查询请求在集群上执行会有哪些步骤？

1.查询请求发送到某个节点
2.该节点收到请求，广播到正这个索引的每个分片上(主分片或副本分片)
3.每个分片执行该请求，并返回结果。
4.节点接收各个分片的结果，合并、排序、返回用户

默认情况下，因为使用 doc_id 进行hash，elasticsearch 也不知道文档的位置，所以必须将查询请求广播到各个分片上。

如果自定义了routing值，使查询更有目的性。不必去广播请求，取而代之的是：文档数据在某个分片上，es直接去这个分片上找。

4. 忘记设置routing

如果在写数据时，忘记写/配置 routing 值，那文档就写到其他分片上去了。

这样子，查询时再用 routing 值就查不到对应数据了。

elasticsearch 提供了另一个功能，在配置mapping时，加入检查 routing 的配置。

即，文档CURD的时候 强制检查，有没有加 routing 参数，如果没有加，就会报错。
