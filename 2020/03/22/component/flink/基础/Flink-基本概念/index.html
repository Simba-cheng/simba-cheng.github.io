    Flink 相关基本概念

    注：部分直接搬运 Apache Flink 零基础入门（一&二）：基础概念解析 中部分内容


    如果有些概念没有看明白，可以先试着写一个 WordCount Demo，等回过头，再理解这些内容。

    Apache Flink 是一个分布式实时处理引擎.

    Flink 应用开发需要先理解 Flink 的Streams、State、Time 基础概念。

    Streams 流

    Stream 流，分为 有限/有界数据流 和 无限/无界数据流

    有限数据流 和 无限数据流 的区别在于：

    无限数据流的数据会随着时间推移而持续增加，持续计算 而且 不存在结束状态；
例如 手机与基站的信号动作、金融股市实时数据等...

有界数据流 数据大小固定，计算最终会完成，并处于结束状态；
例如 HDFS、MySQL、FTP批量、相对固定的数据

    State 状态

    在Flink运行过程中，某些信息会被保存下来，从而可以让 Flink 在发生错误的时候通过这些记录下来的信息恢复到发生错误前的运行状态。

    这些被记录下来的State主要包括三类：

    1.Window收集到的未被处理的数据及Window本身的运行状态

    2.Transformation通过实现Key/Value接口保存的信息

    3.Transformation通过实现CheckpointedFunction接口保存下来的信息

    Time System(时间制度)

    时间分为 Event time(事件事件)、Ingestion time(摄取时间)、Processing time(处理时间)

    Flink 的无限数据流是一个持续的过程，时间 是 我们判断 业务状态是否滞后、数据处理是否及时 的重要依据。

    Event Time 事件时间

    表示事件创建时间，是事件的固有属性，一般会是一条数据的某个字段

    Event Time是每个独立事件在产生它的设备上发生的时间。

    这个时间通常在事件进入Flink之前就已经嵌入到事件中了，并且事件的时间戳是可以从记录中抽取出来的。

    事件时间 常常跟 处理时间 一起使用。

    Ingestion Time 摄取时间

    摄入时间(Ingestion Time)是事件进入Flink的时间，在源操作中每个记录都会获得源的当前时间作为时间戳，后续基于时间的操作(如: time window)会依赖这个时间戳

    摄入时间从概念上来讲是处在事件时间和处理时间之间，与处理时间相比，成本可能会高一点，但是会提供更加可预测的结果。

    因为摄入时间使用的是固定的时间戳(都是在源处指定的)，记录中的不同窗口操作依赖同一个时间戳。

    而在处理时间中每个窗口操作可能将记录赋给不同的窗口(根据本地的系统时钟和传输时延)。

    与事件时间相比，摄入时间程序不能处理任何无序事件或者延迟事件，但是程序无需指定如何产生水印。

    在内部摄入时间和事件时间相似，但是是自动分配时间戳，自动产生水印。

    Processing Time 处理时间

    Processing Time 是指 执行程序 时 对应的 物理机系统时间

    当一个流程序通过处理时间来运行时，所有基于时间的操作(如: 时间窗口)将使用各自操作所在的物理机的系统时间。

    例如:一个每小时处理的时间窗口将包括所有在系统指定的一个小时内到达指定操作的所有记录。

    处理时间是最简单的时间概念，不需要流和物理机之间的协调，它有最好的性能和最低的延迟。

    然而，在分布式或者异步环境中，因为受到记录到达系统时间的影响，处理时间不能够决定系统内操作之间记录流的速度。

    API

    API分为三层，上到下：SQL/Table API、DataStream API、ProcessFunction API，三层。

    API表达能力 以及 业务抽象能力非常强大，但越接近SQL层，表达能力就越弱，抽象能力会增强。

    反之，ProcessFunction 层API表达能力很强，可以进行多种灵活方便的操作，但抽象能力也相对较弱。

    Stream、Transformation、Operator

    开发人员实现的 Flink 程序由 Stream 和 Transformation 这两块构成。

    Stream 是中间数据，而 Transformation 是操作。

    可以将 Transformation 理解成是一个'处理方法'，而 Stream 则是'方法的入参/出参'。

    Transformation 对 ‘入参Stream’ 进行处理/计算，输出 一个或多个’结果Stream’

    一个 Streaming Dataflow 是由一组 Stream 和 Transformation Operator 组成。

    在启动的时候从一个或多个Source Operator开始，结束于一个或多个Sink Operator。


    Source： 代表数据来源，例如：kafka、flie、socket 等…

    Transformation： 数据 转换/处理 等操作。例如：Map、FlatMap、Filter、KeyBy、Window 等…

    Stream： 在各个 Transformation 之间流转的数据。

    Sink： 数据落地，例如：Kafka、DB、Redis、elasticsearch 等…

    Parallel Dataflow

    Flink 程序是 并行和分布式的。

    一个 Stream 可以被分成多个 Stream 分区（Stream Partitions）；

    一个 Operator 可以被分成多个 Operator Subtask，每个 Operator Subtask 在不同的机器或容器中分不同的线程独立运行。

    operator subtasks 的数量在具体的 operator 就是并行计算数，程序不同的 operator 阶段可能有不同的并行数。

    Flink 基本概念

    Flink 是典型的 Master-Slave 架构，各个节点之间通信依靠Akka。

    Flink 实际运行中，包含两类进程，JobManager、TaskManager。

    JobManager

    Master

主要负责分布式集群的协调工作。包括tasks的均衡，checkpoint，以及出错恢复等等。

在Flink中，至少包含一个Job Manager。

有时为了高可用性，会设置多个Job Manager，其中一个是Leader，其他的是Standby。

    TaskManager

    Slave

客户端通过将编写好的 Flink-Job 提交到 JobManager；

JobManager 根据已注册的 TaskManager 的资源情况，将job任务分配给有资源的 TaskManager 节点，然后启动运行

    Client

    Client客户端负责将任务提交到集群，与 JobManager 构建Akka连接；

然后将任务提交到 JobManager ，通过和 JobManager 交互获取任务执行状态。

Client并不参与直接的处理过程。

    Task Slot(任务槽) 是 TaskManager 中的最小资源分配单位，一个TaskManager中有多少个 Task Slot(任务槽) 就意味能支持多少的Task处理。

    能被 Chain 的在一起的 Operator(算子)会放在同一个 Task 中 ，不能被 Chain 在一起的 Operator 会被分隔到不同的Task中。

    Task Slot(任务槽)和资源

    每个 TaskManager(Slave) 都是一个JVM进程，可以在其中执行一个或多个 Task Slot(任务槽)。

    为了控制 TaskManager 接收的任务数量，TaskManager 可以配置 Task Slot(任务槽) 数量(至少一个)。

    每个 Task Slot(任务槽) 代表 TaskManager 中一部分固定资源。

    例如，某个 TaskManager 有三个 Task Slot(任务槽) ，三个 Task Slot(任务槽) 将均分 TaskManager 的内存，用于每个Task Slot(任务槽)。

    注意：CPU资源不会均分、隔离，当前任务槽 只分离内存。


    通过调整任务槽数量，可以自定义子任务如何互相隔离。

    拥有多个任务槽的 TaskManager，意味着多个子任务共享一个JVM。

    同一个JVM中的任务，共享TCP连接(多路复用)和心跳信息，他们还能共享数据集和数据结构。

    默认情况下，Flink 允许子任务共享 任务槽，即使他们是不同任务的子任务，只要他们来自同一个作业。

    数据运算模型

    流处理：只要数据一直在产生，计算就持续地进行

    批处理：在预先定义的时间内运行计算，当完成时释放计算机资源

    WaterMark 水印

    WaterMark 是建立在 EventTime(事件时间) 上的一个概念，用来刻画数据流的完整性。

    如果按照 IngestionTime(摄入时间)、ProcessTime(处理时间) 来衡量，一切都是有序的，完美的，自然而然也就无需watermark了。

    换句话说 EventTime(事件时间) 引入了乱序的问题，而 WaterMark 就是用来解决乱序问题。

    所谓的乱序，其实就是有事件迟到了，对于迟到的元素，我们不可能无限期的等下去，必须要有一种机制来保证一个特定的时间后，必须触发window进行计算了。

    这个特别的机制，就是 WaterMark，它告诉了算子 时间小于等于 WaterMark 的消息不应该再被接收（如果出现意味着延迟到达）。

    也就是 当watermark时间 >= window maxTimestamp时，就符合触发计算的条件了。

    StateBackend

    MemoryStateBackend

    MemoryStateBackend内部把state作为object保存在task manager的java heap中。

    当需要checkpoint时，会将state发送给JobManager，并备份在job manager的heap中。

    显然，这种backend对state的大小有极大的限制，其次可靠性较低。


    FsStateBackend

    FsStateBackend同样把state存储在task manager的heap中。

    当checkpoint时，会持久化到底层的文件系统中。

    因为这种backend持久化state到底层文件系统，所以比MemoryStateBackend要可靠很多， 同时也能提供足够大的存储空间。


    RocksDBStateBackend

    RocksDBStateBackend把state存储在rocksdb (一个轻量的kv database)中。

    当更新state数据时，数据会写在task manager本地的rocksdb里。而checkpoint时， 会将state从rocksdb全部持久化到底层文件系统(hdfs)中。

    rocksdb会将部分数据写在本地磁盘上，大大提高了存储state的容量。

    此外这种backend利用rocksdb LSM tree 写性能优先的特性， 提高了写state性能。

    在flink1.3版本，RocksDBStateBackend还支持增量的checkpoint，可以加快checkpoint的速度，又减少磁盘消耗。
