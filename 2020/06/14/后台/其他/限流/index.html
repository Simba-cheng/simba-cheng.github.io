    缓存、限流、降级 是面对高并发，保护应用系统的三个有效方案。

    缓存： 将接口最终结果集、中间数据结果集，存储一段时间，在 redis cache、local cache 中，提高系统响应速度。

    限流： 通过前期压测，可以大概预知应用系统的性能极限，当 QPS / 并发数 / 请求量 达到某个上限后，会压垮系统。因此需要保证任何时刻 请求量 不能超过阈值，限流就是为了完成这一目的。

    降级： 应用系统调用高峰时（大促、活动），某些不重要的接口，不再对外提供服务；当请求到达时，立即返回，不做业务处理。

    限流算法

    限流算法主要有几种：固定窗口计数器算法、滑动窗口计数器算法、漏桶算法、令牌桶算法。

    固定窗口计数器算法

    *① * 将时间划分为多个窗口；

    *② * 每个窗口内，每有一次请求将计数器加一；

    *③ * 如果窗口计数器超过阈值，超过部分的请求被丢弃。当时间到达下一个窗口时，重置计数器重新计算。


    这个算法存在时间临界问题，即：会让通过的请求量为限制的两倍。

    举个栗子：

    限制窗口时间（秒）最多5个请求。

第一个窗口时间的最后半秒通过了5个请求（20:00:01:500 之后）；
第二个窗口时间的前半秒又通过了5个请求（20:00:02:500 之前）。

时间窗口（秒）：20:00:01:500 - 20:00:02:500 通过了10个请求。

    漏桶算法

    漏桶算法 将所有的请求（水滴）放入漏桶中，漏桶以恒定速度向外’漏出’ 请求（水滴）；

    如果 ‘漏桶’ 空了则停止’漏水’；如果’漏桶’满了，多余的 请求（水滴）直接被丢弃。

    漏桶算法大多数使用队列实现，请求会存入队列中，服务方按照固定的速率从队列中取出请求并执行；超出队列长度的请求直接被拒绝、排队或丢弃。


    漏桶算法强行限制传输速度。当短时间有大量突发请求时，即使此时应用服务没有任何负载，每个请求也需要在队列中带个排队等待一段时间才能被响应。

    令牌桶算法

    对于很多应用场景来说，除了要求能够限制数据的平均传输速率外，还要求允许某种程度的突发传输。

    这时候，令牌桶算法更加合适。

    *① * 以固定速率生成令牌。

    *② * 生成的令牌放入令牌桶中；令牌桶满了多余的令牌直接丢弃。

    *③ * 请求到达时，会尝试从令牌桶中取令牌，取得令牌的请求才会被执行。

    *④ * 如果令牌桶空了，拿不到令牌，那么尝试取令牌的请求会被直接丢弃 或者 阻塞等待直到拿到令牌。

    令牌桶算法既能将所有的请求平均分布到时间区间内，又能承受服务器允许范围内突发的请求。

    Semaphore 信号量

    信号量在操作系统中一般用来管理数量有限的资源。信号量的值表示资源的可用数量。

    在使用资源时，要先从该信号量中获取一个使用许可，成功获取许可之后，信号量可用数量减1。

    在持有许可期间，使用者可以对资源进行操作。完成对资源的使用之后，需要释放许可，信号量可用数加1。

    当信号量可用数为0时，需要获取资源的线程以阻塞的方式来等待资源许可。

    Guava Ratelimiter 限流

    Guava Ratelimiter 可以有效限定单个JVM实例上某个接口的流量，采用令牌桶算法。

    它在并发环境下是线程安全的。不过注意，它不保证公平调用。

    方法 说明
    tryAcquire()

    tryAcquire(int permits)

    tryAcquire(long timeout, TimeUnit unit)

    tryAcquire(int permits, long timeout, TimeUnit unit)    		 从RateLimiter 获取一个令牌，非阻塞

    从RateLimiter 获取指定令牌数，非阻塞

    从RateLimiter 获取一个令牌，如果令牌可以在不超过timeout的时间内获取得到的话，或者如果无法在timeout 过期之前获取得到许可的话，那么立即返回false，非阻塞

    从RateLimiter 获取指定令牌数，如果令牌可以在不超过timeout的时间内获取得到的话，或者如果无法在timeout 过期之前获取得到许可数的话，那么立即返回false ，非阻塞。
    acquire()

    acquire(int permits)    		 从RateLimiter获取一个令牌，该方法会被阻塞直到获取到请求

    从RateLimiter获取指定令牌数，该方法会被阻塞直到获取到请求
    setRate(double permitsPerSecond) 更新RateLimite的速率，参数permitsPerSecond 由构造RateLimiter的工厂方法提供。

    Guava有两种限流模式：

    稳定模式（SmoothBursty）：令牌生成速度恒定，平滑突发限流。

    渐进模式（SmoothWarmingUp）：令牌生成速度缓慢提升直到维持在一个稳定值，平滑预热限流。

    两种模式实现思路类似，主要区别在等待时间的计算上。

    SmoothBursty 模式

    稳定模式，令牌速度生成恒定，平滑突发限流。

    create(double permitsPerSecond)

RateLimiter.create(5);

    令牌桶大小为5，每秒新增5个令牌，即每200毫秒新增一个令牌。（QPS）

    code

    @RestController
public class DemoController {

    private static final Logger LOGGER = LoggerFactory.getLogger(DemoController.class);

    final DateTimeFormatter DATE_TIME_FORMATTER = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss.SSS");

    /**
     * SmoothBursty 模式
     * <p>
     * 桶容量为5，每秒新增5个令牌，即每200毫秒新增一个令牌
     */
    final RateLimiter rateLimiter = RateLimiter.create(5);

    @GetMapping("/sayHello1")
    public String sayHello1() {
        //一次获取一个令牌 非阻塞
        if (rateLimiter.tryAcquire()) {
            LOGGER.info("1---获取到令牌，nowTime:{}", DATE_TIME_FORMATTER.format(ZonedDateTime.now()));
        } else {
            LOGGER.info("没获取令牌（limit）...");
        }
        return "hello";
    }

    @GetMapping("/sayHello2")
    public String sayHello2() {
        //一次获取一个令牌 非阻塞 等待100ms
        if (rateLimiter.tryAcquire(1, 1000L, TimeUnit.MILLISECONDS)) {
            LOGGER.info("2---获取到令牌，nowTime:{}", DATE_TIME_FORMATTER.format(ZonedDateTime.now()));
        } else {
            LOGGER.info("没获取令牌（limit）...");
        }
        return "hello";
    }

    @GetMapping("/sayHello3")
    public String sayHello3() {
        //一次获取一个令牌 阻塞
        rateLimiter.acquire();
        LOGGER.info("3---获取到令牌，nowTime:{}", DATE_TIME_FORMATTER.format(ZonedDateTime.now()));
        return "hello";
    }
}

    SmoothWarmingUp 模式

    渐进模式，令牌生成速度缓慢提升，直到维持在一个稳定值，平滑预热限流。

    permitsPerSecond：每秒新增令牌数。

    warmupPeriod：从冷启动速率过渡到平均速率的时间间隔。

    速率是梯形上升速率的，也就是说冷启动时会以一个比较大的速率慢慢到平均速率；然后趋于平均速率（梯形下降到平均速率）。

    code

    同 SmoothBursty 模式 中code一样，替换 rateLimiter 构造方法即可。

    final RateLimiter rateLimiter = RateLimiter.create(5, 1000, TimeUnit.MILLISECONDS);

    public void output(HttpServletResponse response, String msg) throws IOException {
        response.setContentType("application/json;charset=UTF-8");
        ServletOutputStream outputStream = null;
        try {
            outputStream = response.getOutputStream();
            outputStream.write(msg.getBytes("UTF-8"));
        } catch (IOException e) {
            e.printStackTrace();
        } finally {
            outputStream.flush();
            outputStream.close();
        }
    }
