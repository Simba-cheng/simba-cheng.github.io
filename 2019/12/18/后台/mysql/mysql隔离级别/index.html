    mysql隔离级别

    事务就是一组原子性的SQL查询

    事务内的语句，要么全部执行成功，要么全部执行失败

    ACID

    ACID是数据库事务中的一组属性，即在发生错误/灾难的情况下，也能保证数据的有效性。

    在数据库环境中，满足ACID属性的数据库操作(逻辑操作)称为事务。

    • 原子性(Atomicity)

      一个事务 必须视为不可分割的最小工作单元，整个事务要么全部成功，要么全部失败；不可能只执行其中的一部分。

    • 一致性(Consistency)

      数据库从一个一致性的状态转换到另一个一致性的状态

事务执行前后，数据处于一种合法的状态。

在事务过程中，即便出现异常，因为事务没有最终提交，所以之前做的修改也不会保存到数据库中

    • 隔离性(Isolation)

      一个事务中的操作/修改，在最终提交以前，对其他的事务是不可见的。

并发执行的多个事务，不能相互干扰。

    • 持久性(Durability)

      事务一旦提交，其所做的修改就会永久保存到数据库中。

    隔离级别

    一个事务中的操作/修改，在最终提交以前，对其他的事务是不可见的。

    在SQL标准中定义了四种隔离级别

    级别越低的隔离性可以带来更高的并发，系统开销也越低

注意：每种存储引擎实现的隔离级别都不一样，MySQL-InnoDB 默认是 可重复读（repeatable-read）

    未提交读（read-uncommitted）

    事务中的修改，即使没有提交，对其他事务也是可见的。

也被称为：脏读（Dirty Read）。

    提交读（read-committed）

    一个事务，从开始执行到提交之前，所做的任何修改对其他事务都是不可见的。

一个事务执行过程中，另外一个事务提交了新数据，因此本事务先后两次读到的数据结果会不一致。

也被称为：不可重复读（因为两次执行同样的查询，可能得到不同的结果）

    可重复读（repeatable-read）

    在同一个事务内的查询都是事务开始时刻一致的。

在同一事务中发起多个普通（非锁定）SELECT 语句，其查询结果是相互一致的。

存在幻读问题。

    可串行化（serializable）

    最高级别的隔离级别

通过强制事务串行执行，避免出现幻读。

并发性不好，只有在非常需要确保数据一致和没有并发的情况下，才考虑。

    采用上面四种不同的隔离级别，会带来不同的问题：

    脏读 - 问题

    采用’未提交读’隔离级别，会出现脏读的情况

    一个事务读取到另一个事务未提交的更新数据。

    举个例子：

    A事务正在访问number字段，并对number字段进行了修改，而这个修改还未提交到数据库中（可能会回滚）;

这时B事务也访问了number字段，拿到了A事务还未提交的值。

如果A事务发生了回滚，那B事务中获取number字段的值就是错误的。

    不可重复读 - 问题

    采用’提交读’隔离级别，会出现’不可重复读’的问题

    不可重复读是指：

    在一个事务内，多次读取同一数据，但这个事务还没有结束时；
    如果另一个事务恰好修改了这个数据(并且commit)，那么在第一个事务中，多次读取的数据就可能不一致

    举个例子：

    A事务中，需要查询多次查询a数据

在A事务执行到第一次查询后，B事务更新来a数据(commit)

那么A事务后面两次读取到a的数据和第一次就是不一致的。

    幻读 - 问题

    采用’可重复读’隔离级别，会出现幻读。

    注意：可重复读级别中，重复读只针对同一事务。

    参考《mysql 幻读的详解、实例及解决办法》和《Repeatable Read》中的解释：

    在一个事务中，第一次查询某条记录，发现没有，但是，当试图更新这条不存在的记录时，竟然能成功，并且，再次读取同一条记录，它就神奇地出现了。

    更白话一些：

    幻读 侧重的方面是：

    某一个select操作得到的结果所表现的状态，无法支撑后续业务操作。

    更具体的描述：

    select某条记录是否存在，不存在就插入；但是执行insert的时候，发现这条记录已经存在，无法插入，此时就发生了幻读。

    《mysql 幻读的详解、实例及解决办法》中的栗子：

    step1 T1: SELECT * FROM `users` WHERE `id` = 1;

step2 T2: INSERT INTO `users` VALUES (1, 'big cat'); //并且commit提交事务

step3 T1: INSERT INTO `users` VALUES (1, 'big cat');

step4 T1: SELECT * FROM `users` WHERE `id` = 1;

T1 ：主事务，检测表中是否有 id 为 1 的记录，没有则插入，这是我们期望的正常业务逻辑。
T2 ：干扰事务，目的在于扰乱 T1 的正常的事务执行。

在 RR 隔离级别下，step1、step2 是会正常执行的，step3 则会报错主键冲突，对于 T1 的业务来说是执行失败的，这里 T1 就是发生了幻读；

因为 T1 在 step1 中读取的数据状态并不能支撑后续的业务操作，T1："见鬼了，我刚才读到的结果应该可以支持我这样操作才对啊，为什么现在不可以"。
T1 不敢相信的又执行了 step4，发现和 setp1 读取的结果是一样的（RR下的 MMVC机制）。

此时，幻读无疑已经发生，T1 无论读取多少次，都查不到 id = 1 的记录，但它的确无法插入这条他通过读取来认定不存在的记录（此数据已被T2插入），对于T1来说，它幻读了。

    小结

    不可重复读 和 幻读 的区别

    不可重复读 和 幻读 有时候会搞混，因为两者有些相似。

    不可重复读重点在于：update 和 delete

    幻读的重点在于：insert

    用锁机制来解释的话：

    可重复读中，A事务sql第一次执行读取到数据后，将该数据加锁，其他事务无法修改这些数据，以此实现可重复读。

但这种方法无法锁住 insert 的数据。

所以当事务A先读取来数据，或者修改来全部数据；事务B还是可以 insert 提交数据，

这时候事务A就会莫名其妙的多了一条前面没有的数据

这就是幻读；

它不能通过"行级锁"来避免。
